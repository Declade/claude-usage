# Phase 1: Scaffold - Research

**Researched:** 2026-03-11
**Domain:** macOS SwiftUI app project creation — MenuBarExtra, entitlements, Xcode project scaffold from CLI
**Confidence:** HIGH

## Summary

Phase 1 creates the Xcode project from scratch (no project file exists yet) and establishes the three architectural foundations that, if wrong at project inception, require a full rewrite to fix: the correct `MenuBarExtra` style, disabled App Sandbox, and no Dock icon. All three must be locked in this phase before any business logic is written.

The recommended project-creation method is **XcodeGen** (`brew install xcodegen` + `project.yml`). This produces a reproducible, git-friendly Xcode project from a YAML spec and avoids the `.xcodeproj` merge conflict problem. The alternative — opening Xcode and using the GUI wizard — produces a correct project but adds a binary-diff `.xcodeproj` to the repo with no CLI-reproducible audit trail. For a personal tool with no team, either works; XcodeGen is strictly better for the planning/execution workflow.

The deployment target decision (macOS 13 vs 14) must be made in this phase because it determines whether `@Observable` is available throughout the codebase. Evidence from TelemetryDeck (February 2026) shows macOS 26 at 39.5% and Sequoia/15 declining — together with Sonoma/14, the macOS 14+ share is well above 90% of active developer machines. The correct decision is **macOS 14.0** as the deployment target, enabling `@Observable` throughout with no fallback complexity.

**Primary recommendation:** Use XcodeGen with `project.yml` to generate the Xcode project. Set deployment target to macOS 14.0. Configure `LSUIElement = YES` in Info.plist inline and `com.apple.security.app-sandbox = false` in the generated entitlements file. Use `MenuBarExtra(.window)` style. Verify App Sandbox is disabled against an **archive** build, not just debug.

### Pattern 1: XcodeGen project.yml for macOS Menu Bar App
**What:** A YAML spec that drives XcodeGen to produce a correctly-configured `.xcodeproj`
**When to use:** At project creation; regenerate after any structural change to targets or settings

```yaml
# Source: XcodeGen ProjectSpec.md (github.com/yonaskolb/XcodeGen/blob/master/Docs/ProjectSpec.md)
name: ClaudeUsage
options:
  bundleIdPrefix: com.marcschuelke
  deploymentTarget:
    macOS: "14.0"

targets:
  ClaudeUsage:
    type: application
    platform: macOS
    deploymentTarget: "14.0"
    sources: [ClaudeUsage]
    info:
      path: ClaudeUsage/Info.plist
      properties:
        LSUIElement: true
        CFBundleDisplayName: Claude Usage
    entitlements:
      path: ClaudeUsage/ClaudeUsage.entitlements
      properties:
        com.apple.security.app-sandbox: false
    settings:
      base:
        PRODUCT_BUNDLE_IDENTIFIER: com.marcschuelke.claudeusage
        SWIFT_VERSION: "6.0"
        MACOSX_DEPLOYMENT_TARGET: "14.0"

  ClaudeUsageTests:
    type: bundle.unit-test
    platform: macOS
    deploymentTarget: "14.0"
    sources: [ClaudeUsageTests]
    dependencies:
      - target: ClaudeUsage
```

### Pattern 4: Deployment Target Decision
**What:** macOS 14.0 as the minimum deployment target
**Rationale:** macOS 26 is at 39.5% and Sequoia/15 declining (TelemetryDeck Feb 2026); macOS 14+ is ~90%+ of developer machines. macOS 14 enables `@Observable` throughout all phases, eliminating `@Published`/`ObservableObject` boilerplate. This is a personal tool — the Ventura-only slice (~5-10%) is acceptable to exclude.

### Anti-Patterns to Avoid
- **Leaving App Sandbox enabled:** Sandbox blocks `~/.claude/` reads in non-debug builds silently. Must be disabled in the entitlements file, not just a build setting.
- **Testing only in debug scheme:** The sandbox setting behaves differently in debug vs archive builds. Verify with an archive build (`Product > Archive` or `xcodebuild archive`).
- **Using `.menuBarExtraStyle(.menu)` for panel content:** The `.menu` style strips most SwiftUI controls and only supports text/buttons/dividers in menu-style rendering. Progress bars, sliders, and custom views require `.window` style.
- **Not setting `LSUIElement`:** Without it, the app appears in the Dock AND the App Switcher. Setting it in Info.plist makes the app an "agent" — menu bar only, no Dock presence.
- **Calling `NSApp.terminate` from a Button inside `.menu` style:** Works in `.window` style but has different behavior in menu-style rendering.
- **Importing `AppKit` for terminate:** Use `NSApplication.shared.terminate(nil)` which is accessible from SwiftUI via `import AppKit` — but no AppKit scaffolding or `AppDelegate` is needed for this phase.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Xcode project file | Manual `.xcodeproj` XML editing | XcodeGen from `project.yml` | `.xcodeproj` is a binary-adjacent format; conflicts are painful; XcodeGen is the standard solution |
| App Sandbox configuration | Custom build script | `entitlements.properties` in `project.yml` | XcodeGen generates the `.entitlements` file correctly; manual entitlements have signing-order bugs |
| Menu bar presence | `NSStatusBar` + `NSStatusItem` + `NSPopover` | `MenuBarExtra(.window)` | The AppKit pattern requires ~80 lines of boilerplate; MenuBarExtra is canonical on macOS 13+ |
| Quit functionality | Custom quit logic | `NSApplication.shared.terminate(nil)` | Standard macOS app termination; handles all cleanup hooks correctly |

**Key insight:** Everything in Phase 1 is configuration and boilerplate, not logic. The value is getting the configuration exactly right before business logic is written — not implementing anything novel.

## Common Pitfalls

### Pitfall 1: App Sandbox Silently Blocking File Reads
**What goes wrong:** App reads `~/.claude/projects/**/*.jsonl` in debug (Xcode run), data appears, everything looks fine. After archiving or distributing, the app reads nothing. No error is thrown — `FileManager` returns empty or `String(contentsOf:)` throws a permission error that is swallowed.
**Why it happens:** Xcode's default new macOS app template enables App Sandbox. Debug builds in Xcode are codesigned with development entitlements that are more permissive. Archive builds enforce sandbox strictly.
**How to avoid:** In `project.yml`, set `com.apple.security.app-sandbox: false` in `entitlements.properties`. After generating the project and building, verify with `codesign -d --entitlements - path/to/ClaudeUsage.app` that the sandbox key is absent or false.
**Warning signs:** Debug works, archive does not. `FileManager.default.fileExists(atPath:)` returns false for paths under `~/.claude/` in non-debug builds.

### Pitfall 2: Dock Icon Appearing on First Launch
**What goes wrong:** App appears in Dock and App Switcher despite `LSUIElement` being set. Happens intermittently on first launch of a freshly-built app.
**Why it happens:** `LSUIElement` must be in the compiled Info.plist that is embedded in the .app bundle, not just in an Xcode build setting. If using XcodeGen's `info.properties`, this is handled correctly. If set via Xcode's Info tab, verify the key name is `LSUIElement` (Boolean), not `Application is agent (UIElement)` (string).
**How to avoid:** Use `info.properties: LSUIElement: true` in `project.yml`. Verify with `defaults read path/to/ClaudeUsage.app/Contents/Info.plist LSUIElement` returning `1`.
**Warning signs:** App icon flashes in Dock on launch, then disappears. App appears in Cmd+Tab switcher.

### Pitfall 3: Wrong MenuBarExtra Style Locked In
**What goes wrong:** Using `.menuBarExtraStyle(.menu)` (the default) produces a pull-down native menu. Progress bars, `ProgressView`, custom SwiftUI layouts — none of these render in `.menu` style. They are silently ignored or shown as empty space.
**Why it happens:** The default `MenuBarExtra` style is `.menu`, which maps content to `NSMenu` items. Only `Button`, `Text`, `Divider`, `Link`, and `Toggle` are supported.
**How to avoid:** Always add `.menuBarExtraStyle(.window)` to the `MenuBarExtra` scene in Phase 1. This is a scene-level modifier that cannot be changed per-view later.
**Warning signs:** `ProgressView` shows nothing. `VStack` with custom views renders as a blank menu. Only `Button` and `Text` appear.

### Pitfall 4: MenuBarExtra Has No Programmatic Close
**What goes wrong:** No API to programmatically dismiss the panel (tracked as FB11984872). Tapping outside dismisses it, but calling `.isPresented` or any close method from within the view does nothing.
**Why it happens:** Known SwiftUI limitation — `MenuBarExtra(.window)` does not expose the underlying `NSPanel` or a Binding that controls visibility.
**How to avoid:** Accept this limitation in Phase 1. If programmatic close is critical (e.g., "Quit" should close panel first), use `MenuBarExtraAccess` SPM package which provides a Binding workaround. For Phase 1, `NSApp.terminate(nil)` from Quit is fine — it quits the whole app.
**Warning signs:** Trying to add `@Environment(\.dismiss)` inside MenuBarExtra content and finding it has no effect.

### Pitfall 5: Swift 6 Concurrency Warnings on First Build
**What goes wrong:** Setting `SWIFT_VERSION: "6.0"` generates `@MainActor`-isolation warnings on any code that touches UI from a non-main-actor context. Phase 1 scaffold likely has none, but later phases will.
**Why it happens:** Swift 6 strict concurrency mode is the correct default for this app (safe, future-proof), but requires consistent `@MainActor` annotation on App/View types and `await MainActor.run {}` for background-to-main dispatch.
**How to avoid:** Keep Swift 6 mode. In Phase 1, `@main struct ClaudeUsageApp: App` is implicitly `@MainActor`. No concurrent code in this phase, so no warnings expected.
**Warning signs:** Build fails with "Sending value of type across actor boundaries" in later phases — not Phase 1 itself.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `NSStatusBar` + `NSStatusItem` + `NSPopover` | `MenuBarExtra(.window)` in SwiftUI | macOS 13 / WWDC22 | Eliminates ~80 lines of AppKit boilerplate |
| `SMLoginItemSetEnabled` | `SMAppService` | macOS 13 / WWDC22 | Login item registration; old API deprecated |
| `@Published` + `ObservableObject` | `@Observable` macro | macOS 14 / WWDC23 | Simpler state, finer-grained view updates, less boilerplate |
| Xcode GUI project creation | XcodeGen from `project.yml` | Community practice | Reproducible, vcs-friendly, avoids xcodeproj binary conflicts |
| `generate-xcodeproj` via SPM | XcodeGen or direct Xcode | Swift 5.5+ | SPM's `generate-xcodeproj` is deprecated and removed |

**Deprecated/outdated:**
- `NSStatusBar`/`NSStatusItem`/`NSPopover` pattern: Superseded by `MenuBarExtra` on macOS 13+; still works but requires AppKit imports and boilerplate
- `SMLoginItemSetEnabled`: Deprecated; `SMAppService` is the replacement (Phase 4)
- `swift package generate-xcodeproj`: Deprecated and removed from Swift toolchain

## Open Questions

1. **Bundle identifier ownership**
   - What we know: `com.marcschuelke.claudeusage` is used in examples above
   - What's unclear: Is this the intended bundle ID? Affects code signing.
   - Recommendation: Planner should confirm or parameterize in `project.yml` — easy to change before first build

2. **Menu bar icon asset**
   - What we know: `systemImage: "chart.bar.fill"` works as a Phase 1 placeholder
   - What's unclear: Whether a custom icon asset is desired for Phase 1 or deferred to polish phase
   - Recommendation: Use system image in Phase 1; custom icon is a Phase 4 polish item

3. **Hardened Runtime requirement**
   - What we know: `ENABLE_HARDENED_RUNTIME: YES` is required for notarization; irrelevant for personal builds
   - What's unclear: Whether notarization is ever desired for this tool
   - Recommendation: Enable it anyway (no cost, prevents future friction); it does not restrict file access when sandbox is off
